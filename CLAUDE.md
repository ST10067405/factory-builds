# Factory Builds — deployment notes

This repo is deployed as a **single Vercel project**. Each top-level folder (e.g.
`dimitri-pet-services/`, `tipsy-flamingo/`) is a self-contained concept site for one
customer, and each customer must only be able to reach their own site.

## How site isolation works

- `slugs.json` maps each folder name to a random, unguessable slug (e.g.
  `dimitri-pet-services -> "a1b2c3d4e5f6..."`). This file is the durable source of
  truth for every customer link — **never rename, remove, or regenerate an existing
  entry**, since that breaks a link already sent to a real customer.
- `scripts/build.js` is the Vercel `buildCommand`. It reads `slugs.json`, mints a
  slug for any folder that doesn't have one yet, copies every folder's contents
  verbatim into `dist/<slug>/`, and writes a blanket `dist/robots.txt`
  (`Disallow: /`). The business-name path (e.g. `/dimitri-pet-services/`) never
  exists in the deployed output — only `/<slug>/` does.
- `dist/` is the build output, is **gitignored**, and should never be edited or
  committed directly — it's fully regenerated by `scripts/build.js` every build.

## Adding a new concept site

1. Add the new folder at repo root (same shape as existing folders: an `index.html`
   plus an assets folder, all internal links/asset paths relative).
2. Run `node scripts/build.js` **locally** — this mints and writes the new slug into
   `slugs.json`.
3. Commit the updated `slugs.json` and push **before** triggering a Vercel deploy.
   If a brand-new folder is deployed without first committing its slug, Vercel's
   build container mints one that never makes it back into git — the next deploy
   would then mint a *different* slug and silently break the link already shared
   with that customer.
4. Look up the new folder's slug in `slugs.json` and share `https://<vercel-domain>/<slug>/`
   with that customer.
